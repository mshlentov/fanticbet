# Правила разработки FanticBet

Документ описывает конвенции проекта. Соблюдение обязательно — как для человека, так и для ИИ-ассистента (Z Code).

## 2. Коммиты — Conventional Commits

Формат: `<тип>: <описание на русском>`

### Типы

| Тип | Когда использовать |
|---|---|
| `feat:` | Новая функциональность (новый эндпоинт, сервис) |
| `fix:` | Исправление бага |
| `refactor:` | Рефакторинг без изменения поведения |
| `docs:` | Только документация (README, docs/, комментарии в markdown) |
| `test:` | Добавление/изменение тестов |
| `chore:` | Сборка, зависимости, конфиги (без логики) |
| `style:` | Форматирование, отступы (без изменения кода) |
| `perf:` | Оптимизация производительности |

### Правила

- Заголовок ≤ 72 символов
- После пустой строки — тело с деталями (опционально)
- Одна логическая изменяемая единица = один коммит
- Ссылка на задачу из `tasks.md` в теле: `M1 — Авторизация и кошелёк`

### Примеры

```
feat: M1 — регистрация пользователя с бонусом

- AuthService.Register в одной транзакции создаёт user + wallet
- Начисление signup_bonus через wallet_transactions
- Валидация email/пароля через validator/v10
```

```
fix: гонка при параллельных ставках на один кошелёк

Добавлен SELECT ... FOR UPDATE в PlaceBet. Теперь два параллельных
запроса не могут увести баланс в минус.
```

```
docs: правила разработки (conventions)
```

## 3. Git — стратегия ветвления

### Ветви

- `main` — всегда собирается, всегда работает. Только через PR.
- `feat/M<n>-<кратко>` — ветки вех/фич (например `feat/M1-auth`)
- `fix/<кратко>` — ветки багфиксов (например `fix/wallet-race`)
- `chore/<кратко>` — ветки для рутины (например `chore/update-deps`)

### Процесс

1. Создать ветку от актуального `main`:
   ```bash
   git checkout main && git pull
   git checkout -b feat/M1-auth
   ```
2. Делать атомарные коммиты в ветке
3. Запушить и открыть PR в GitHub
4. После проверки (self-review или ИИ-review) — squash merge в `main`
5. Удалить ветку после слияния

### Когда можно коммитить прямо в `main`

- Мелкие правки документации (`docs:`)
- Правки опечаток
- Обновление `.env.example`

## 4. Архитектура слоёв

Строго: `handler → service → repository`. Запрещено:

| Запрет | Почему |
|---|---|
| Handler вызывает repository напрямую (минуя service) | Бизнес-логика утекает в HTTP-слой |
| Service знает про `*gin.Context`, HTTP-статусы | Слой не должен зависеть от транспорта |
| Repository содержит бизнес-правила (проверки, расчёты) | Репозиторий — только CRUD и запросы |
| Domain-модель используется как DTO запроса | Разделяем внутреннюю модель и внешний API |

### Импорты в Go-файле — 3 группы, разделённые пустой строкой

```go
import (
    // 1. Стандартная библиотека
    "context"
    "fmt"

    // 2. Внешние пакеты
    "github.com/gin-gonic/gin"
    "github.com/jackc/pgx/v5"

    // 3. Внутренние пакеты проекта
    "fanticbet/internal/config"
    "fanticbet/internal/domain"
)
```

## 5. Работа с ошибками

- **Без `panic`** в бизнес-логике. Ошибки возвращаются явно: `if err != nil { return err }`.
- **Без `_ = err`** — все ошибки обрабатываются.
- При логировании ошибки добавляем контекст: `log.Printf("PlaceBet: user_id=%d outcome_id=%d: %v", ...)`.
- В handler — единый формат ответа:
  ```json
  { "error": { "code": "validation_error", "message": "..." } }
  ```
- В service/repository ошибки оборачиваем: `fmt.Errorf("PlaceBet: %w", err)`.

### Коды ошибок

| Код | HTTP | Когда |
|---|---|---|
| `validation_error` | 400 | Неверный запрос (валидация DTO) |
| `unauthorized` | 401 | Нет/неверный токен |
| `forbidden` | 403 | Не админ, нет прав |
| `not_found` | 404 | Ресурс не существует |
| `conflict` | 409 | Дубликат, гонка |
| `insufficient_balance` | 400 (или 409) | Недостаточно фантиков |
| `market_closed` | 409 | Рынок закрыт, ставка невозможна |
| `internal_error` | 500 | Всё остальное |

## 6. Context.Context

- **Первый параметр** всех функций, работающих с БД/HTTP/внешними сервисами: `ctx context.Context`.
- В handler — берётся из `c.Request.Context()` (Gin прокидывает контекст запроса).
- В воркерах — создаётся с `context.WithTimeout` для каждой итерации.
- **Запрещено** `context.Background()` в handler/service (кроме инициализации и тестов).

---

## 7. Деньги и числа

| Величина | Тип | Почему |
|---|---|---|
| Баланс, ставка, payout, bonus | `int64` | Фантики — целые. Никаких float для денег |
| Коэффициенты | `pgtype.Numeric` или `decimal` | NUMERIC(8,3) в БД |
| `potential_payout` | `floor(stake * odds)` через decimal | Без потерь точности |

**Все операции с кошельком — в транзакции с `SELECT ... FOR UPDATE`** на строке `wallets`.

---

## 8. Транзакции и идемпотентность

- Кошелёк, размещение ставки, settlement — **всегда** в одной DB-транзакции.
- `FOR UPDATE` обязателен при чтении баланса перед изменением.
- Settlement **идемпотентен**: обрабатываем только `pending`-ставки, повторный прогон безопасен.
- В каждой транзакции запись в `wallet_transactions` (append-only журнал).

## 9. Тесты

- Файлы `_test.go` рядом с тестируемым кодом (`user.go` → `user_test.go`).
- Табличные тесты — предпочтительный формат:
  ```go
  tests := []struct{
      name string
      input int
      want int
  }{ ... }
  ```
- Имена: `TestFunctionName_Scenario`, например `TestPlaceBet_InsufficientBalance`.
- Repository тестируется через testcontainers или test-БД (не моки GORM).
- Service тестируется с мок-репозиториями (интерфейсы).
- **Критичные места обязательны с тестами**: кошелёк, `PlaceBet`, settlement.

---

## 10. Миграции

- Каждая миграция имеет `up` и `down` (откатимость обязательна).
- Имя: `NNNNNN_<кратко>.up.sql` / `.down.sql` (например `000003_create_bets`).
- Одна миграция = одно логическое изменение.
- **Никогда** не правим применённую миграцию — пишем новую.

---

## 11. DTO и валидация

- В handler — отдельные структуры запросов/ответов, не domain-модели:
  ```go
  type RegisterRequest struct {
      Email      string `json:"email" binding:"required,email"`
      Password   string `json:"password" binding:"required,min=8"`
      DisplayName string `json:"display_name" binding:"required,min=2"`
  }
  ```
- Валидация через `github.com/go-playground/validator/v10` (теги `binding:`).
- Никаких неявных полей — все поля в DTO явно.

## 12. Frontend (React + TS)

| Что | Правило |
|---|---|
| Имена компонентов | PascalCase (`EventCard.tsx`) |
| Имена хуков | `useAuth`, `useBets` |
| Имена типов | PascalCase, явно (`type User = {...}`) |
| Стили | Tailwind или выбранный UI-кит (без inline-style для логики) |
| Запросы | TanStack Query (без `useEffect + fetch`) |
| Состояние авторизации | Context + useAuth хук |
| API-клиент | Один файл `web/src/api/client.ts` с Bearer-инъекцией и auto-refresh |

---

## 13. Логирование

- Сейчас `log.Printf` (стандартная библиотека). Позже — slog/zerolog (M7).
- Формат: `[func] ключ=значение: %v`. Контекст обязательный.
  ```go
  log.Printf("PlaceBet: user_id=%d outcome_id=%d stake=%d: %v",
      userID, outcomeID, stake, err)
  ```
- Никаких логов паролей, токенов, персональных данных.

## 14. Работа с ИИ-ассистентом (Z Code)

- ИИ следует этим правилам так же, как человек.
- Если ИИ нарушил правило — указать в комментарии и попросить исправить.
- Перед каждой вехой/задачей — ИИ читает `docs/architecture.md`, `docs/tasks.md`, `docs/conventions.md`.
- После завершения задачи — коммит с префиксом вехи (`feat: M1 — ...`).
- Большие изменения — через PR, ИИ делает self-review по этому чек-листу.

### Чек-лист self-review перед коммитом

- [ ] Комментарии и коммит-сообщение на русском
- [ ] Идентификаторы на английском
- [ ] Соблюдены слои (handler → service → repository)
- [ ] Импорты сгруппированы (stdlib / external / internal)
- [ ] Все ошибки обработаны (`if err != nil`)
- [ ] `context.Context` первый параметр
- [ ] Деньги в int64, без float
- [ ] Операции с кошельком в транзакции с `FOR UPDATE`
- [ ] Критичные места покрыты тестами
- [ ] Миграция имеет `up` и `down`
- [ ] DTO с тегами валидации, без domain-моделей в запросах
